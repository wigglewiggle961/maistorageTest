## 1. System Classification

**System Type:** RAG (Evaluation Harness)

**Description:**
A locally-running evaluation system designed to measure the performance of an Agentic RAG system. It compares the agentic retrieval loop (with self-correction) against a naive baseline using standardized RAGAS metrics. The system must run offline using Ollama models to ensure data privacy and zero cloud cost.

**Critical Failure Modes:**
1. **Bias in Evaluation:** The judge model (`gemma4:e4b`) incorrectly scoring its own outputs due to lack of diversity in perspectives.
2. **Metrics Misinterpretation:** Reporting aggregate scores that mask significant per-question failures.
3. **Hardware Overload:** Evaluation process exceeding the 6GB VRAM limit by attempting to load multiple models simultaneously.
4. **Data Leakage:** Inadvertently using evaluation ground truth during the agentic retrieval/generation steps.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling / Cloud Infrastructure Documentation

**User Population:** Developers and Cloud Architects seeking accurate information about Azure AI Foundry Agent Service.

**Stakes Level:** Medium (Information retrieval for technical setup; error could lead to deployment delays or misconfiguration).

**Output Consequence:** Correctness of technical implementation and architectural understanding of Azure AI Foundry.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|--------------------|--------|--------|
| Technical Accuracy | Directions and parameters match latest documentation exactly. | Obsolete parameters or incorrect command syntax. | High | Azure AI Foundry Docs |
| Citations | Every claim is traced back to a specific documented section. | Uncited claims or hallucinations not present in docs. | High | INGEST-03 |
| Completeness | Answer covers all parts of the user question. | Partial answers that miss critical setup steps. | Medium | RAG-05 |

### Known Failure Modes in This Domain

- **Version Confusion:** Mixing up "Basic" and "Standard" setup requirements.
- **Resource Naming:** Using names of discontinued or renamed Azure services.
- **Context Exclusion:** Prioritizing FAQ summary over detailed technical guide context.

### Regulatory / Compliance Context

None identified for this local prototype assessment.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Technical Reviewer | Verifying that ground truth in `eval_dataset.json` is faithful to source docs. |
| AI Evaluator | Running the RAGAS harness and interpreting the delta between agentic and baseline systems. |

## 2. Framework Decision

**Selected Framework:** RAGAS

**Version:** Latest (0.2.x)

**Rationale:**
RAGAS is the industry standard for reference-free and reference-based RAG evaluation. It provides standardized metrics (Faithfulness, Relevancy, Precision, Recall) and integrates natively with LangChain components, making it ideal for the existing Python/LangChain/Ollama stack.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| DeepEval | Heavier abstraction, less focus on the RAG specific metrics compared to Ragas. |
| TruLens | Better for tracing/debugging (v2), but Ragas is superior for producing the quantitative assessment report required for the deliverable. |

**Vendor Lock-In Accepted:** No (Open-source, local execution).
